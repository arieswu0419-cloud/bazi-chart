# Aries 八字分析網站

純前端靜態網站，學員輸入姓名／性別／出生日期時間，計算四柱、五行、十神、大運、流年，並可點選大運／流年查看對應神煞。帳號需管理者審核後才能使用。

版面、欄位順序與配色都比照 [metagroup-bazi-ming-juan](https://www.event.meta-academy.biz/metagroup-bazi-ming-juan) 實際取樣調整：四柱欄位為「時／日／月／年」由左到右，大運與流年也改成跟四柱一樣的寬表格（十神／天干／地支／藏干／長生／神煞），點選欄位標題整欄反白，由老到少排列，並自動依今天日期反白目前所在的大運／流年。主色為金色 `#C9B872`、頁面底色為米色 `#EBE7DB`，取樣自參考網站本身的顏色。命盤結果可一鍵匯出成 PDF（含 Aries LOGO，大運流年會另起一頁）。

## 一、建立 Firebase 專案（帳號審核系統）

GitHub Pages 只能放靜態檔案，帳號密碼與審核狀態需要 Firebase 來處理，一樣是免費額度。

1. 前往 [Firebase Console](https://console.firebase.google.com/)，建立新專案。
2. 左側選單「Build > Authentication」→ 開始使用 → 登入方式選「電子郵件/密碼」→ 啟用。
3. 左側選單「Build > Firestore Database」→ 建立資料庫 → 選正式環境模式（production mode）、選一個離您近的地區（如 asia-east1）。
4. 進入 Firestore 的「規則」分頁，把本專案 `firestore.rules` 的內容整份貼上取代，然後發布。
5. 左側選單「專案設定」（齒輪圖示）→ 一路往下滑到「您的應用程式」→ 選網頁圖示 `</>` 新增網頁應用程式 → 註冊後會看到一段 `firebaseConfig = {...}` 設定值。
6. 把這段設定值複製貼到 `js/firebase-config.js`，取代裡面的 `YOUR_API_KEY` 等預設文字。

## 二、審核學員帳號＋權限管理（管理者操作）

**帳號＝信箱、密碼由使用者自訂，三個報告（八字／人格解碼／生命靈數）各自有獨立的開關權限**，
由系統維護帳號（寫死在 `js/auth.js` 的 `ADMIN_EMAIL` 常數，目前是 `arieswu0419@gmail.com`）登入
`admin.html` 管理，不用再逐筆手動改 Firestore：

- **核准新申請＋開通權限**：用系統維護帳號登入網站後，右上角會出現「系統維護」連結（一般帳號看不到這個連結）。
  進去之後每個帳號一列，點狀態按鈕在「審核中／已核准」間切換，勾選八字／人格解碼／生命靈數三個checkbox 就是即時
  開通對應報告的權限，不用存檔按鈕，每次勾選都會馬上寫回 Firestore。
- 使用者剛申請帳號時，`permissions` 三個報告預設全部是 `false`（關閉），核准之後還要記得勾選需要開放的報告，
  不然學員登入後會看到「您的帳號目前沒有任何報告的使用權限」的提示，三個報告按鈕/頁籤都不會顯示。
- 這個功能上線前就已經 `approved` 的舊帳號，Firestore 文件裡沒有 `permissions` 欄位，前端會當作「沿用舊行為、
  三個報告都開放」處理（`js/app.js` 裡 `data.permissions || {bazi:true,renge:true,lifenum:true}` 這個 fallback），
  不會因為新功能上線就把舊學員鎖住；但只要系統維護帳號在 `admin.html` 幫該帳號勾選過一次（即使全勾），
  Firestore 就會寫入明確的 `permissions` 物件，之後就照這個物件的值走，不會再套用 fallback。
- **停用某帳號**：目前仍然要到 Firebase 主控台 Authentication 分頁 → 找到該使用者 → 右側選單「停用帳戶」
  （這個跟「審核狀態」是兩件事：停用是完全擋登入，審核狀態/權限只是能不能看到報告內容）。
- **第一個管理者帳號**：直接到 `signup.html` 用 `arieswu0419@gmail.com` 申請帳號，申請完還是要先照 Firebase
  主控台的方式把自己的 `status` 改成 `approved`（管理者帳號的「審核狀態」跟「是不是系統維護帳號」是分開判斷的，
  審核狀態沒過一樣不能登入），之後才能用這個帳號登入 `admin.html`。

**重要：改過 `firestore.rules` 之後一定要重新發布**——這份規則檔只是這個 repo 裡的原始碼，改完存檔
不會自動生效在真正的 Firebase 專案上，一定要照本節「一、」的步驟 4，把整份新內容貼到 Firebase 主控台
Firestore Database → 規則分頁 → 發布，管理權限（`isAdmin()`）跟權限欄位的檢查才會真的開始生效；
發布前 `admin.html` 會因為 Firestore 規則還是舊版而讀取失敗（顯示「Missing or insufficient permissions」）。
這個環境沒有安裝 Firebase CLI，沒辦法用指令自動部署規則，只能手動貼到主控台。

## 三、部署到 GitHub Pages

1. 在 GitHub 建立一個新的 repository（例如 `aries-bazi`）。
2. 把這個資料夾整包推上去：
   ```
   git init
   git add .
   git commit -m "Aries 八字分析網站"
   git branch -M main
   git remote add origin https://github.com/你的帳號/aries-bazi.git
   git push -u origin main
   ```
3. GitHub 上該 repo 的 Settings → Pages → Source 選 `main` 分支、`/ (root)` 目錄 → Save。
4. 幾分鐘後會產生網址，例如 `https://你的帳號.github.io/aries-bazi/`。

## 四、本機測試

直接雙擊 `index.html` 用瀏覽器打開即可（Firebase 設定填好之後）。若遇到瀏覽器安全限制導致 Firebase 連不上，可以改用簡易本機伺服器再開啟，例如：

```
python -m http.server 8000
```
再到瀏覽器打開 `http://localhost:8000/login.html`。

## 與參考網站／課程講義的比對結果

除了用瀏覽器實際操作 [metagroup-bazi-ming-juan](https://www.event.meta-academy.biz/metagroup-bazi-ming-juan) 背後的計算頁面逐項核對，也讀了 `補充/` 資料夾裡的課程講義（Meta Academy《八字命卷》系列 PDF），裡面第二卷第五章「十神命格判斷法」「十神特別格局」直接給了命格判斷的具體規則與範例，據此重寫了格局判斷邏輯；五行強弱表／十神比重表的加權公式則是拿好幾組不同生日的數字回推校正出來的（課程講義沒有寫出明確公式）。

**已逐項核對一致**：
- 四柱天干地支、五行、陰陽正負號
- 藏干內容與**顯示順序**（12 個地支的實際排列順序逐一測出來寫成對照表，不是猜的公式——多數地支是 lunar-javascript 原生順序反過來，但巳／未／申三支是另一種排法）
- 十神（天干、藏干）、十二長生
- **五行強弱表**：公式是「4 個天干 + 4 個地支的本氣」各算 1 分，中氣／餘氣不計分，總分 8，跟參考網站測試的多組數字完全一致
- **十神比重表**：天干（不含日主自己）算 1 分，地支本氣算 0.63 分、中氣／餘氣各算 0.2 分，子午卯酉這種只有一個藏干的地支算 1 分（等同天干）——這組權重是回推校正出來的，跟參考網站數字非常接近（多數落在四捨五入誤差內）
- **命格判斷**：改成課程講義教的邏輯——月支十二長生為「臨官」判建祿格、「帝旺」判羊刃格；否則看月支藏干本氣是否透干（出現在任一天干，含日干月干自己），本氣不透才看中氣／餘氣。多組測試案例（含課程講義本身的範例）判斷結果一致
- 天干合／克、地支三合／三會／六合／六沖的文字提示
- 生肖與年柱都用「立春」交界（精確到分鐘）判斷——測試了 2000 年立春交界前後（20:30 vs 20:45），年柱和生肖確實剛好在交界那一刻切換
- **大運／流年歲數**：改成跟參考網站一樣的「西曆實歲」（出生那一年＝0歲），原本用的是虛歲（出生那一年＝1歲），已修正並對照多組流年年份確認一致；大運也修正為固定抓 10 組（原本少抓了 2 組最年長的）
- 大運／流年現在跟四柱一樣，點開會顯示完整的十神／天干／地支／藏干／長生／神煞

**神煞（第二輪比對，讀了 `再補充/` 資料夾《八字解鎖篇》第十六、十七章的神煞專章後全部重寫）**：

原本的神煞是自己拼湊的坊間版本，跟參考網站對不上（例如月柱／年柱神煞完全不同）。《八字解鎖篇》第十七章逐頁列出每個神煞的正式對照表（天乙貴人、太極貴人、福星貴人、文昌貴人、天官貴人、國印貴人、學堂貴人、天德貴人、月德貴人、龍德、紅鸞天喜、桃花咸池、紅艷煞、孤辰寡宿、流霞、羊刃飛刃、血刃、劫煞、災煞、亡神、歲破大耗、五鬼、白虎、天狗、喪門、病符、天醫、金輿、祿神、魁罡貴人、十靈、陰差陽錯），照著逐條抄錄重寫後，用 1976-04-19 20:20（女）這組資料實測，**四柱每一柱的神煞內容都跟參考網站逐字一致**。過程中也發現並修正了幾個重要問題：

- **魁罡是「四柱有即算」**：原本誤以為只檢查日柱，實際上時柱／月柱／年柱只要干支剛好是庚辰、庚戌、壬辰、戊戌都算，不限日柱。
- **福星貴人有些天干對應兩個地支**（乙對應丑、亥；丙對應子、戌），原本只寫了一個，漏掉一半。
- **華蓋、將星等三合局神煞會有「自己對自己」的假命中**：地支剛好落在辰戌丑未（庫位）時，用該支自己的三合局去查，會查到自己，這種情況不算數，已修正排除。
- **年空／空亡的顯示**：四柱的地支旁邊會分別對照「日柱所在旬」與「年柱所在旬」，命中哪一組空亡就標示「空」或「年空」，現在跟參考網站一樣會顯示出來。

**已知還沒解決的一個案例**：測試 1990-05-18 14:30（男）這組資料時，命格算出「正印格」，但參考網站顯示「正官格」——月支巳的藏干本氣（丙）不透干，中氣（庚）有透干（年干庚），照課程講義規則應該用中氣定格（正印格才對），但參考網站卻是用餘氣（戊）判定為正官格。已經深入比對多組資料、也讀了課程講義相關章節，還是沒找到能同時解釋這個案例又不跟其他案例衝突的規則，可能牽涉到更進階的判斷（例如身強身弱的取捨），暫時保留這個已知差異，其餘測試案例的命格都判斷正確。

## 神煞第三輪測試（20 筆立春／節氣交界 + 20 筆一般神煞比對）

先用 20 筆卡在立春、其餘 24 節氣交界前後 3 分鐘的資料測試，年柱／生肖（立春切割）與月柱（節氣切割）**20 筆全部跟參考網站一致**，證實分鐘級的交界判斷是準確的。

接著測試共 20 筆一般神煞比對（分兩批進行），發現並修正：

- **原本標成「學堂貴人」的那組數值，實際上是天廚貴人**，已改名為 `TIAN_CHU`。逐一測試 10 個日干後：乙／己／庚／辛／癸五干數值正確；甲、戊兩干原值錯誤，已用實測結果改正（甲：亥→巳；戊：寅→申）；丙、壬兩干原值其實是學堂貴人的值，已從天廚貴人表移除（避免掛錯名字），天廚貴人在這兩干的正確值還沒確認；丁干尚未測到足夠資料，暫時保留原值（可能不準確）。
- **新增真正的學堂貴人**（`XUE_TANG_REAL`）：測試中確認了 6 個日干的數值（丙：寅、丁：寅、戊：申、己：申、辛：巳、壬：申），甲／乙／庚／癸四干還沒測到，暫時不加入。
- **新增六秀日**：固定 6 組日柱干支（丙午、丁未、戊子、戊午、己丑、己未），比照魁罡貴人／十靈的查表方式，測試中丙午、丁未、戊子三組都已比對確認一致。
- **修正血刃年支對照表（`XUE_REN`）**：丑、戌兩個年支原值錯誤，已用實測結果改正（丑：酉→巳；戌：子→卯）。
- **修正病符年支對照表（`BING_FU`）**：年支為酉時，原值（申）會讓網站多顯示一個參考網站沒有的病符，正確值還沒確認，已先移除這一格避免誤判。

## 神煞第四輪測試（20 筆，針對金匱／詞館／天醫／學堂剩餘天干／十靈）

針對第三輪留下的問題再測 20 筆（其中 10 筆刻意挑選十靈清單裡的 10 組日柱組合各測一次），發現並修正：

- **十靈清單裡的「庚戌」是誤判**：逐一測試清單裡全部 10 組日柱組合後，只有庚戌這組參考網站沒有顯示十靈，其餘 9 組（甲辰、乙亥、丙辰、丁酉、戊午、庚寅、辛亥、壬寅、癸未）都測試確認一致，已把庚戌從清單移除。
- **天廚貴人（`TIAN_CHU`）再修正庚干**：原值（巳）測試後發現不對，已改正為亥；同時確認壬干原值（申）其實是學堂貴人的值，已改正為天廚貴人真正的值（寅，這是新測到的）。
- **學堂貴人（`XUE_TANG_REAL`）新增甲、乙、庚三干**：甲：亥、乙：亥、庚：巳，都各有 2 筆資料交叉驗證一致。加上上一輪的丙丁戊己辛壬，10 個天干只剩癸還沒測到。

**已知還沒解決、這輪測試發現比原本想的更複雜的部分**：

- **金匱貴人、詞館貴人依然沒有實作**（後續第六、七輪已分別解出，詳見下方）。這輪測試發現金匱貴人的規則比想像中複雜：原本以為是單純「以日干查地支」，但同樣是辛干，不同命盤測到的地支不一樣（一次是酉、一次是卯），代表金匱貴人可能同時受日干、月干或其他因素影響，不是單一天干查表就能決定，需要用更嚴謹的方式（固定其他變數、只變動單一天干）重新測試才能找出真正規則，暫時不實作以免顯示錯誤內容。詞館貴人目前只有 2 筆資料（壬：戌、癸：酉，皆年干查、不比對日柱本身），資料仍然太少。
- **血刃在年支自我對照時可能有遺漏**：測試中發現年支剛好是亥（血刃對照值也是亥）時，參考網站的年柱本身也會標血刃，但目前程式為了避免其他神煞的「自我對應」問題，統一把年柱排除在血刃等年支查表之外，這部分還沒有單獨處理。
- **XUE_REN（血刃）、BING_FU（病符）其餘年支值**只針對測試中發現有問題的項目做了修正，其餘尚未逐一測完，如果之後測試又發現對不上，可能還有其他年支的值不準確。

## 神煞第五輪測試（52 筆，系統性隔離變數找金匱／詞館／天醫規則）

針對金匱、詞館、天醫三個神煞，改用「控制變數」的方式重新設計測試（固定其他條件、只變動單一變數），總共測了 52 筆：

## 神煞第九輪：血刃「巳月見巳」是誤測，四柱本身也要驗證（不能只靠大運流年資料）

使用者提供兩筆比對資料（莊振豪 1982-07-26 18:00 男、何姵璇 1982-05-30 10:45 女），逐一核對四柱神煞。莊振豪（月支未）四柱神煞全部吻合；何姵璇（月支巳）的**時柱跟月柱都被多標了一個「血刃」**，參考網站兩柱都沒有。

原因：第八輪的 `XUE_REN_MONTH`（血刃的月支查表）裡有「巳：巳」這一格，屬於「月支自我對應」。用何姵璇這筆資料實測兩次（時支分別排到巳、申），月柱（巳）在參考網站上都沒有顯示血刃，證實這格是錯的——第八輪當時可能是從大運流年表反推規則時，剛好把別的神煞跟血刃搞混了（大運流年的神煞欄位很密集，逐字比對時容易看錯行）。這輪先把「巳：巳」從 `XUE_REN_MONTH` 移除，改標未確認；第十輪再用何姵璇的大運流年表找到正確值「巳：酉」，已補上，見下方。

這輪也順便確認了「年空」的顯示邏輯本身沒有問題：一開始用純文字（`get_page_text`）比對參考網站的四柱時，誤以為「年空」標在時柱的巳上，但實際截圖比對後發現「年空」其實標在日柱（丑）上——文字版面是線性排列，容易看錯欄位歸屬，之後這類需要精確判斷「標記在哪一柱」的比對，要用截圖而不是純文字確認。

**教訓**：像 `XUE_REN_MONTH` 這種「大量透過大運流年反推出來的規則」，之後應該再挑幾筆命盤本身（四柱）剛好落在該規則的資料驗證一次，不能只信任大運流年比對的結果——大運流年神煞欄位密集、資料量大，比對時比四柱本身更容易看錯，尤其是自我對應（比如「巳見巳」）這種容易跟旁邊其他神煞搞混的情況要特別小心。

## 網頁快取問題：更新網站後記得清除瀏覽器快取

使用者回報十神比重表數字跟參考網站對不上（例如正印顯示 5.7% 但參考網站是 2.9%），比對後發現**目前的程式碼算出來的數字其實跟參考網站完全一致**（重新用同一筆資料試算，十神比重表 10 個數字全部吻合），問題出在 `index.html` 等頁面原本引用 `js/bazi-engine.js` 等本機檔案時沒有加版本參數，瀏覽器會把舊版本的檔案快取起來，即使 GitHub 上的程式碼已經更新，瀏覽器還是繼續用快取的舊版本執行，導致看到的是幾輪修正之前的結果。

已經把 `index.html`、`login.html`、`signup.html` 裡所有本機的 `.css`／`.js` 檔案都加上 `?v=2` 版本參數強制瀏覽器抓新版。**之後每次更新網站程式碼，都要把這個版本號往上加一**（例如下次改成 `?v=3`），否則使用者（包含自己測試時）還是會看到快取的舊版本。如果測試時發現數字看起來像是「舊的」，可以先按 Ctrl+F5（強制重新整理，略過快取）確認是不是快取問題，再回報是不是真的程式邏輯有誤。

**已逐項核對一致**：
- **天賦數／主命數**：生日「年月日」全部數字加總得到原始總和，如果總和是兩位數以上就逐位相加直到個位數。天賦數＝顯示原始總和以及所有中間的兩位數結果（用「/」分隔，例如「37/10」），主命數＝最後那個個位數；天賦數的總和本身就是個位數時，天賦數跟主命數顯示同一個數字。
- **年數能量（用「實歲」的擇日年，不是選單選的西元年本身）**：`= reduce( reduce(生日月) + reduce(生日日) + reduce(有效擇日年) )`。**有效擇日年＝如果擇日的月/日還沒到生日的月/日，要當作還沒過生日，擇日年要減 1 才算對**（跟真實年齡「還沒過生日不算長一歲」同一個道理，生日當天算已經過生日）。這個規則是額外測試發現的：原本以為年數能量只看擇日年份本身，因為 3 份 PDF 剛好都是「生日在擇日之前」（擇日固定 11/29，PDF 三人生日都在 3～4 月），測不出這條規則；換一組生日在擇日之後的資料（例如生日 12 月、擇日 11 月）才發現算出來的年數能量對不上，才抓出「有效擇日年」這個規則，10 筆新測試資料全部驗證一致。
- **月／日數能量**：對稱公式，月數能量 `= reduce( reduce(生日年) + reduce(生日日) + reduce(擇日月) )`，日數能量 `= reduce( reduce(生日年) + reduce(生日月) + reduce(擇日日) )`（reduce＝反覆逐位相加到個位數）。這兩個沒有像年數能量那樣的「有效年」問題，已用大範圍調整擇日月/日/年分別測試過，兩者都只跟各自對應的擇日分量有關，不受生日/擇日先後影響。
- **人生階段能量表**：4 個欄位對應 60-79／40-59／20-39,80-99／0-19 四個年齡區間，是逐一疊加出生時間分量的 digitSum：60-79 欄＝年，40-59 欄＝年＋月，20-39/80-99 欄＝年＋月＋日（跟天賦數同一個總和），**0-19 欄＝年＋月＋日＋時**——這一欄需要「出生時辰」才能算，沒填時辰時這一欄顯示「需填出生時間」，不會顯示假數字。這是直接在計算頁上，固定同一筆生日只改「時」欄位（0/8/14/19），對照結果變化（10/1 → 18/9 → 15/6 → 20/2）逐一反推出來的。
- **空缺數／九宮連線密碼共用的「數字池」**：1~9 之中，沒有出現在這個池子裡的數字就是空缺數；九宮連線密碼的 12 條線（含 4 條兩格連線）也是看這條線的數字是否全部出現在同一個池子裡才算命中。池子的組成：**生日年/月/日本身的數字＋「日」的 digitSum（只有日，不含年/月各自的 digitSum）＋天賦數的化簡過程（天賦數是個位數、沒有化簡過程時改用主命數頂替）＋目前歲數對應的那個階段欄（跟九宮能量圖橘色用的是同一欄，見下方）＋年數能量本身**。這條規則是額外測試 10 筆資料才逐步修正出來的，過程中發現兩個原本用 3 份 PDF 測不出來的錯誤：(1) 原本以為生日「年」「月」各自的 digitSum（Yraw/Mraw）都要算進池子，但測到一筆生日年份的 digitSum 沒有被生日本身數字覆蓋的資料（1988-06-20，Yraw=digitSum(1988)=26 中的「2」「6」都不在生日原始數字裡）才發現 Yraw/Mraw 不能算，只有 Draw（日）要算；(2) 原本以為固定用「20-39/80-99」那一欄（因為天賦數剛好等於這一欄），但測到一筆現場歲數落在 40-59 欄的資料（1965-12-3）才發現要跟九宮能量圖橘色一樣，用「目前歲數對應的階段欄」，不是固定用某一欄。
- **九宮能量圖**：1-9 每個數字一個直立堆疊色塊，依照主命數（藍）／天賦數（粉）／生日數（綠）／階段數（橘）／年數能量（紫）5 個類別疊加，每個類別的數字裡每出現一次某個位數就疊一格：
  - 藍＝主命數本身的數字；紫＝年數能量本身的數字；粉＝天賦數的化簡過程（跟上面數字池用的是同一個規則，天賦數是個位數時改用主命數頂替）
  - 綠＝生日年/月/日原始數字，**再加上「日」單獨化簡到個位數過程中的中間值**（例如日＝19 要經過 19→10→1 兩步，中間的 10 跟 1 都要算進去；日＝22 只要一步 22→4，只多算一個 4；日本身就是個位數則不額外加）——這個規則是拿吳慧琳（日＝19，兩步化簡）這筆資料的綠色格子數量對不上才抓出來的
  - 橘＝目前歲數對應的階段欄（跟上面數字池用的是同一欄），只疊歲數所在那一欄（0-19/20-39/40-59/60-79/80-99，80-99 跟 20-39 同一欄），不是 4 欄全疊；0-19 欄即使沒填時辰，這裡疊色塊時一律當時辰＝0 計算（跟人生階段能量表格子本身「需填出生時間」不顯示假數字是兩條分開的邏輯）
  - 13 筆資料、5 類別×9 個數字逐一核對全部吻合。
